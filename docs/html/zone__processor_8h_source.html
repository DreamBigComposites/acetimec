183 //---------------------------------------------------------------------------
184 // Externally exported API. The workflow is roughly:
185 //
186 // 1) Initialize the AtcZoneProcessor upon memory allocation. This needs to be
187 // done only once. This is essentially the "constructor" of this object.
188 // 2) Initialize the AtcZoneProcessor with the AtcZoneInfo. This will usually be
189 // done once for the duration of the client application. However, it can be
190 // called more than once if the AtcZoneProcessor is reused (to save memory) for
191 // multiple zone infos. The transition cache (see below) will be invalidated.
192 // 3) Initialze the AtcZoneProcessor with the epoch seconds or LocalDateTime
193 // to populate the transition cache for that year.
194 // 4) Find the AtcFindResult for the epoch seconds or AtcLocalDateTime of
195 // interest using the cache of transitions.
196 //
197 // The AtcTimeZone object and its associated `atc_time_zone_xxx()` functions
198 // know how to follow the above workflow, and client applications will normally
199 // use the functions attached to the AtcTimeZone object.
200 //---------------------------------------------------------------------------
