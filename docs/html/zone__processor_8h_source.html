190 //---------------------------------------------------------------------------
191 // Externally exported API. The workflow is roughly:
192 //
193 // 1) Initialize the AtcZoneProcessor upon memory allocation using
194 // `atc_processor_init()`. This needs to be done only once. This is essentially
195 // the "constructor" of this object.
196 //
197 // 2) Initialize the AtcZoneProcessor with the AtcZoneInfo. This will usually be
198 // done once for the duration of the client application. However, it can be
199 // called more than once if the AtcZoneProcessor is reused (to save memory) for
200 // multiple zone infos. The transition cache (see below) will be invalidated.
201 //
202 // 3) Initialze the AtcZoneProcessor with the epoch seconds or LocalDateTime
203 // to populate the transition cache for that year.
204 //
205 // 4) Find the AtcFindResult for the epoch seconds or AtcLocalDateTime of
206 // interest using the cache of transitions.
207 //
208 // The AtcTimeZone object and its associated `atc_time_zone_xxx()` functions
209 // know how to follow the above workflow, and client applications will normally
210 // use the functions attached to the AtcTimeZone object.
211 //---------------------------------------------------------------------------
